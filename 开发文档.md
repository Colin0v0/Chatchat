# Chatchat 开发文档

更新时间：2026-03-29

## 1. 项目概览

Chatchat 是一个前后端分离的聊天应用，当前核心能力包括：

- 多模型聊天：支持 `Ollama` 与 `OpenAI 兼容接口`
- 工具增强：支持 `RAG` 与 `Web Search`
- 推理展示：支持 `Thinking` / `Reasoning Trace`
- 多模态输入：支持图片，以及统一附件链路下的文件输入
- Markdown 渲染：支持代码块、表格、Mermaid 等富文本展示

当前整体原则：

- 后端按领域拆分，不把所有逻辑继续堆回 `main.py`
- `RAG` 与 `Web Search` 统一收敛到 `retrieval/`
- 附件统一走本地解析链路，不做“原生文件直传模型”和“本地解析”双轨并存
- 尽量不写 fallback 逻辑，优先保持边界清晰、行为可预期

## 2. 根目录结构

```text
Chatchat/
  backend/
  frontend/
  storage/
  docker-compose.yml
  README.md
```

说明：

- `backend/`：FastAPI + SQLAlchemy + 多模态/检索/模型接入
- `frontend/`：React + TypeScript + Vite
- `storage/`：宿主侧持久化目录，主要给容器和本地运行共用
- `docker-compose.yml`：当前容器编排入口

## 3. 前端架构

前端源码目录：

```text
frontend/src/
  app/
  components/
  lib/
  types.ts
  App.tsx
```

### 3.1 主要职责

- `App.tsx`
  - 页面总装配
  - 侧边栏、主区域、设置弹窗拼装

- `app/useChatApp.ts`
  - 前端核心状态中心
  - 负责会话列表、当前会话、输入草稿、模型选择、发送/重试/停止
  - 当前是前端最关键的业务 hook

- `app/useResponsiveSidebar.ts`
  - 负责桌面端/移动端侧边栏展开收起行为

- `app/useComposerAttachments.ts`
  - 统一管理输入框附件草稿
  - 支持图片预览与普通文件草稿

- `app/useConversationStreams.ts`
  - 独立管理流式会话状态
  - 负责 `AbortController`、流式 stage、占位 assistant message、停止恢复输入等逻辑

- `app/chatSessionUtils.ts`
  - 聊天会话相关纯函数
  - 负责临时消息构造、附件恢复、流式 stage 映射、会话摘要更新

- `components/`
  - 输入框、消息列表、侧边栏、设置、Markdown 渲染等 UI 组件

- `lib/api.ts`
  - 前端 API 请求封装
  - 负责普通请求与流式聊天请求

### 3.2 当前输入区设计

当前输入区已经从“图片专用”切到“通用附件输入”：

- `+` 按钮直接打开文件选择
- 支持文件类型：
  - 图片
  - PDF
  - Markdown / TXT / 代码 / JSON / HTML / XML / YAML
  - CSV / XLSX
  - DOCX
- 支持把图片或文件直接拖进对话框
- 用户消息操作区新增“回填到输入框”按钮
  - 当前只回填文本
  - 不回填历史附件

### 3.3 当前流式事件协议

前端当前消费的流式事件类型：

- `meta`
- `status`
- `reasoning`
- `token`
- `sources`
- `done`
- `error`

这套协议是前后端联动的关键，不要轻易改字段名。

### 3.4 当前前端会话流设计

这一轮前端已经把“聊天状态”和“流式执行状态”做了拆分：

- `useChatApp.ts`
  - 管理页面级状态与业务动作
  - 负责发起新消息、重试、切会话、模型切换、附件草稿

- `useConversationStreams.ts`
  - 管理运行中的流式会话
  - 负责：
    - 临时 user/assistant message 插入
    - `meta` 事件后把临时会话 id 替换成真实会话 id
    - `status -> stage` 映射
    - 停止生成后把文本与附件恢复回输入框

- `chatSessionUtils.ts`
  - 作为纯函数层，承接 message/sources/attachment 的结构转换
  - 避免把细碎会话操作继续堆回 `useChatApp.ts`

  llm/
  multimodal/
  retrieval/

### 4.1 目录职责

- `api/`
  - FastAPI 路由层
  - 当前包括聊天、会话、模型、RAG 相关接口

- `chat/`
  - 聊天工作流
  - 包括消息历史准备、会话上下文、流式事件输出、重试逻辑

- `core/`
  - 基础配置
  - 当前主要是 `config.py`

- `llm/`
  - 模型注册、能力发现、Ollama / OpenAI 兼容接口接入

- `multimodal/`
  - 图片理解、附件解析
  - 当前图片与文件输入都在这一层收口

- `retrieval/`
  - 检索统一域
  - 包含显式检索计划、RAG、websearch

- `storage/`
  - 数据库、ORM、媒体存储

- `main.py`
  - 应用装配入口
  - 只负责创建 `FastAPI`、挂载路由、静态资源和 startup 初始化

### 4.2 main.py 约束

`backend/app/main.py` 目前刻意保持很薄：

- 创建 `FastAPI`
- 挂载 `/media`
- 注册 CORS
- startup 时执行 `Base.metadata.create_all()` 与 `ensure_schema()`
- 把 `build_chat_services(settings)` 放到 `app.state`

这里不应该继续堆业务逻辑。

## 5. 检索与工具架构

### 5.1 当前统一策略

检索相关逻辑全部收口到 `retrieval/`：

```text
retrieval/
  plan.py
  service.py
  strategy.py
  rag/
  websearch/
```

这样做的原因：

- `RAG` 和 `Web Search` 本质上都是“上下文获取工具”
- 检索计划和 context merge 应该只维护一套
- 避免未来工具扩展时出现平行架构

### 5.2 当前工作流程

一轮聊天大致流程：

1. 前端发送消息或附件
2. 后端创建用户消息
3. `chat/streamer.py` 输出 `meta`
4. `MessageHistoryService` 准备消息历史
5. 如有附件，先补附件上下文
6. `retrieval/plan.py` 根据前端显式选择的 `retrieval_mode` 构建检索计划
7. `retrieval/service.py` 拉取上下文并聚合来源
8. 调用 `llm/` 开始流式生成
9. 逐步输出 `reasoning` / `token`
10. 输出 `sources` / `done`

### 5.3 当前工具状态展示

前端会根据流式事件展示：

- `status`
- `thinking trace`
- `sources`

当前 `status` 会被前端映射成有限的几个 stage：

- `Reading image` -> `analyzing_attachments`
- `Reading notes` -> `reading_notes`
- `Searching` -> `searching`

这一层已经基本稳定，后续新增工具时优先复用现有事件模型，不再恢复旧的 `tool_plan` 分支。

### 5.4 当前检索策略约束

当前检索计划已经从“模型 planner 推断”收缩成“前端显式模式 + 后端显式计划”：

- `retrieval_mode=none`
  - 不做检索，直接回答

- `retrieval_mode=rag`
  - 走 `NOTE_FIRST`
  - 优先使用笔记/RAG 结果

- `retrieval_mode=web`
  - 走 `WEB_FIRST`
  - 优先使用联网搜索结果

`RetrievalService` 负责：

- 生成 `RetrievalPlan`
- 并行执行对应检索
- 聚合 `sources`
- 按策略给不同来源打权重
- 生成注入模型的 `context_message`

## 6. 多模态与附件设计

### 6.1 当前策略

附件统一走本地解析链路，不再区分：

- 原生文件输入模型链路
- 本地解析链路

当前统一后的好处：

- 前端体验一致
- 后端行为稳定
- 不需要针对不同模型维护两套文件工作流

### 6.2 当前支持范围

#### 图片

- 走图片理解链路
- 本地视觉模型 + OCR
- 生成 `attachment_context`

#### 文件

当前已接入本地解析入口：

- `PDF`
  - 使用 `pypdf`

- `Markdown / TXT / 代码 / JSON / HTML / XML / YAML`
  - 统一按文本解析

- `CSV / XLSX`
  - 统一走表格摘要解析
  - 当前输出列头和前若干行样本

- `DOCX`
  - 提取段落与表格文字

### 6.3 当前附件端到端链路

当前附件链路已经是统一实现，前后端都不再区分“图片专用输入”和“文件专用输入”：

1. 前端 `ChatComposer`
   - 通过 `input[type=file]` 或拖拽接收附件
   - 交给 `useComposerAttachments.ts` 生成草稿

2. 前端发送
   - `useChatApp.ts` 读取 `draftAttachments`
   - `lib/api.ts` 用 `FormData` 把 `message`、`files`、`model`、`retrieval_mode` 一起发到 `/api/chat/stream`

3. 后端落盘
   - `storage/media.py` 校验数量、大小、类型
   - 根据类型落到 `storage/media/images/...` 或 `storage/media/files/...`
   - 返回 `StoredAttachment`

4. 后端建模
   - `chat/workflow.py` 创建 `Message`
   - `chat/context.py` 把附件写入 `message_attachments`

5. 流式准备阶段
   - `chat/streamer.py` 调用 `MessageHistoryService`
   - 如果消息里有附件但还没有 `attachment_context`，就先触发附件解析

6. 附件解析阶段
   - `multimodal/attachment.py` 负责聚合图片与文件
   - 图片走 `ImageTextService`
   - 文件走 `FileParser`
   - 最终产出统一的 markdown `attachment_context`

7. 注入模型阶段
   - `MessageHistoryService` 把用户原始文本与 `Machine-generated attachment brief` 合并
   - 这份文本既用于聊天模型输入，也用于检索 query/history 的补全

8. 前端展示阶段
   - 用户消息显示 `MessageAttachmentStrip`
   - 输入框草稿显示 `ComposerAttachmentStrip`
   - 停止生成时可以把文本和附件一起恢复回输入区

### 6.4 当前多模态后端文件

关键文件：

- `backend/app/multimodal/image.py`
- `backend/app/multimodal/file_types.py`
- `backend/app/multimodal/file_parser.py`
- `backend/app/multimodal/attachment.py`

职责划分：

- `image.py`：图片理解
- `file_types.py`：附件类型识别
- `file_parser.py`：文件解析
- `attachment.py`：图片与文件上下文聚合

### 6.5 当前附件上下文缓存策略

附件解析不是每次都重新跑。

- `MessageHistoryService._ensure_attachment_context()`
  - 优先读取 `messages.attachment_context`
  - 没有缓存时才触发解析
  - 解析结果写回数据库

- `messages.image_context`
  - 目前仍保留旧字段
  - 当消息里有图片且旧字段为空时，会同步写入，主要用于兼容旧图片链路

## 7. 数据与存储设计

### 7.1 当前数据库表

当前核心表：

- `conversations`
- `messages`
- `message_attachments`

### 7.2 关键字段

#### messages

- `content`
- `image_context`
- `attachment_context`
- `sources_json`

说明：

- `image_context` 是旧字段，为了兼容已有图片链路保留
- `attachment_context` 是新字段，统一保存附件上下文

#### message_attachments

- `kind`
  - 当前可取 `image` / `file`
- `original_name`
- `mime_type`
- `relative_path`
- `size_bytes`
- `extension`
- `position`

### 7.3 当前存储位置

- SQLite：`backend/storage/app.db`
- 媒体文件：`backend/storage/media/`
- 图片文件：`backend/storage/media/images/YYYY/MM/DD/`
- 普通文件：`backend/storage/media/files/YYYY/MM/DD/`
- RAG 索引：`backend/storage/rag/index.json`

### 7.4 schema 演进方式

当前没有上 Alembic。

数据库变更依赖：

- `Base.metadata.create_all()`
- `storage/database.py` 中的 `ensure_schema()`

这意味着：

- 小规模字段新增目前可接受
- 如果后面表结构继续复杂化，建议上正式 migration

## 14. 开发约定

当前项目默认遵循这些约定：

- 优先做清晰重构，不堆补丁
- 不主动引入 fallback 分支
- 新功能优先收口到已有领域目录
- 前后端接口字段改名时，尽量一次性收干净，不留双命名并存

如果后面继续重构，优先保持这三件事：

1. 单一职责清晰
2. 数据流可追踪
3. 前后端协议稳定
